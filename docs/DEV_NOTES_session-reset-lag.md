# Dev Note — Session 5h โชว์ 0% + "resets in" หาย หลังทิ้งไว้ข้ามคืน

**วันที่:** 2026-07-16
**อาการที่รายงาน:** ทิ้ง app ไว้ตั้งแต่ 22:00 มาดูเช้า 09:03 เห็น Session (5h) = 0%,
บรรทัด "resets in" หายไป, ใช้ AI ต่อแล้วเลขไม่ขึ้น (ทั้งที่ usage จริง ~10%), กด Refresh เองถึงกลับมา

## สรุป: **ไม่ใช่บั๊ก** — เป็น latency ของ poll ชนจังหวะ window รีเซ็ตพอดี

อ่าน `%APPDATA%\ClaudeMeter\logs\claude-meter-2026-07-16.log` → poll ทำงาน**ทุก 3 นาทีตลอดคืน
ไม่มี error ไม่มี gap เครื่องไม่ได้ sleep** ลำดับเหตุการณ์ช่วงเช้า:

| เวลา | Session 5h | เกิดอะไร |
|------|-----------|----------|
| 08:58:26 | 2% | poll ปกติ |
| **09:01:26** | **0%** | 5h window รีเซ็ตจริง → API คืน `resets_at: null` |
| 09:01:52 | 0% | เปิด popup (auto-refresh on open) — ยัง 0% เพราะเพิ่งรีเซ็ต + ยังไม่ได้ใช้ AI |
| 09:04:20 | 10% | กด Refresh เอง (หลังใช้ AI) |
| 09:04:26 | 11% | poll อัตโนมัติรอบถัดไป (จะได้ 11% อยู่แล้วโดยไม่ต้องกด) |

## ทำไมเป็นแบบนั้น

1. **"resets in" หาย = ถูกต้องตามข้อมูล** — ตอน Session = 0% window ไม่ active,
   API ส่ง `resets_at: null` มา ([UsageClient.cs:119](../src/UsageClient.cs#L119),
   [PopupForm.cs:520](../src/PopupForm.cs#L520) วาด "resets in" เฉพาะเมื่อ `ResetsAt` มีค่าและยังไม่หมด)
   → ตอน 0% ไม่มี window ให้รีเซ็ต จึงไม่มีเวลาให้โชว์ (5h window รอบใหม่จะเริ่มนับเมื่อใช้ AI ครั้งถัดไป)

2. **"ใช้ AI แล้วไม่ขึ้น" = poll latency ≤3 นาที** — app poll ทุก 180 วินาที
   ([TrayAppContext.cs:70](../src/TrayAppContext.cs#L70)) เพราะ endpoint rate-limit หนักถ้าเร็วกว่านี้
   ตอนคุณใช้ AI (09:01–09:04) usage ใหม่จะเข้าตอน poll รอบถัดไป (09:04:26) — คุณกด Refresh เอง
   เร็วกว่า 6 วินาที (09:04:20) เลยเห็นก่อน แต่ถึงไม่กดมันก็อัปเดตเองใน 3 นาที

## ทำไมมันดู "ค้าง" ทั้งที่ไม่ค้าง
เผอิญมาดูจอ**พอดีช่วงหลัง window รีเซ็ต (0%) ก่อน poll รอบถัดไป** — เจอ 0% + ไม่มี resets in
พร้อมกัน เลยดูเหมือนตาย แต่จริงๆ มันจะหายเองใน ≤3 นาที

## ต้องแก้โค้ดไหม
**Logic ไม่ต้องแก้** — poll เร็วกว่า 180s ไม่ได้ (rate limit) พฤติกรรมถูกต้องแล้ว

**ทำ UX เสริม (แล้ว):** ตอน Session (5h) = 0% + ไม่มี `resets_at` เดิมเว้นว่าง → เปลี่ยนเป็นโชว์
`"resets 5h after next use"` ชิดขวาแทน (ตำแหน่งเดียวกับ "resets in …")
- [PopupForm.cs](../src/PopupForm.cs) — `DrawRow` (สาขา `else if (w.Key == "five_hour")`),
  เพิ่มความกว้างใน `ComputeWidth`, ค่าคงที่ `NextUseHint`
- แค่ช่วยให้เข้าใจง่ายตอนเจอจังหวะหลังรีเซ็ต ไม่ได้แก้ latency (ซึ่งไม่ใช่บั๊ก)

## ที่ยืนยันแล้วว่า **ไม่ใช่** สาเหตุ
- timer ค้าง / `_fetching` ค้าง → log มี poll ครบทุก 3 นาที ตัดออก
- เครื่อง sleep แล้ว timer ไม่ฟื้น → log ต่อเนื่องไม่มี gap ตัดออก
- rate-limit backoff ค้างยาว → ไม่มี error ใน log เลย ตัดออก
